## Codebase Patterns
- Module manifests use `claudeos-module.json` with fields: name, version, description, main, requires
- All module entry points are `src/index.ts` (loaded via tsx at runtime, no compilation)
- The kernel server at `kernel/server.ts` handles HTTP, WebSocket, and module loading
- Module loader in `kernel/module-loader.ts` discovers from `./modules/` dir and `node_modules/@claude-nix-os/`
- Docker image is `node:22-slim` with tsx installed globally, runs as root for volume access
- Railway deployment: `railway up --detach` from repo root, healthcheck at `/`
- Next.js config at `next.config.ts` uses `serverExternalPackages: ['ws']`
- Zustand for state management, individual selectors to avoid re-render loops
- The kernel auto-generates JWT_SECRET and AUTH_TOKEN if not provided via env vars
- **Proxy routing**: kernel server routes `/` → OpenVSCode Server (port 8000), `/api/*` → Next.js, `/ws` → ClaudeOS WebSocket
- **Process management**: supervisord manages openvscode-server (priority 100) and kernel-server (priority 200)
- **OpenVSCode settings**: Stored in /data/.openvscode/User/settings.json, applied via init-openvscode.sh on container start
- **Task scheduling pattern**: Use EventEmitter + setTimeout for scheduled execution, load pending tasks on startup, clean up timers on dispose
- **Natural language time parsing**: Regex-based parser for "in X minutes/hours/days", "tomorrow at X", "next Monday at X"
- **Session spawning**: HTTP POST /api/sessions → WebSocket connection with JWT auth → send message content
